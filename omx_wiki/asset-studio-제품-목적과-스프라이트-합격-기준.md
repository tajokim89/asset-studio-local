# Asset Studio 제품 목적과 스프라이트 합격 기준

> 2026-07-11 정정: 과거의 “`dungeon-cleanup-inc` 전용 제작 도구”라는 전제는 폐기한다. 최신 제품·구조 결정은 [[asset-studio-최신-제품-방향-및-생성-검증-재설계]]를 따른다.

## 확정된 제품 목적

Asset Studio는 **범용 도트 게임 에셋 생성·경량 편집 프로그램**이다.

핵심 작업 흐름은 다음과 같다.

    생성 레시피 선택
      -> AI 후보 여러 개 생성
      -> 자동 검증과 비교
      -> 사용자 승인
      -> 캔버스에서 소폭 수정
      -> 출력 프로필에 맞춰 내보내기

포토샵 수준의 범용 편집기를 다시 만드는 것이 목적은 아니다. AI가 쓸 만한 초안을 반복 가능하게 만들고, 사용자가 마지막 결함만 빠르게 고칠 수 있게 하는 것이 목적이다.

`dungeon-cleanup-inc`는 전용 목표가 아니라 다음 용도로만 사용한다.

- 초기 출력 프로필 예제
- 실제 게임 연동 회귀 fixture
- 품질 평가용 프로토타입 소비자
- UI, 스프라이트, 파일 구조 계약의 사례 연구

다른 게임과 장르도 별도 출력 프로필로 지원할 수 있어야 한다.

## Provider 결정

- 기본 생성 경로는 **Hermes 에이전트**로 둔다.
- 유료 직접 API는 사용자가 명시적으로 설정했을 때만 선택 가능한 보조 경로로 둔다.
- Asset Studio가 Hermes 내부 모듈을 직접 import하지 않고, 자체 Provider 인터페이스 뒤에서 호출한다.
- Hermes가 없어도 편집, 프로젝트 열기, 기존 이미지 QA와 내보내기는 동작해야 한다.

## 현재 핵심 실패

현재 가장 큰 문제는 이미지 생성 성공을 제품 성공으로 취급한다는 점이다.

- 한 번의 요청으로 완성 시트 한 장을 기대한다.
- idle, walk, attack 같은 이름은 있지만 실제 동작 의미를 검증하지 않는다.
- 후보 비교 전에 자동으로 캔버스에 채택한다.
- QA 코드가 생성·채택 차단 조건으로 연결되지 않았다.
- 기술 설정과 미완성 subtype이 기본 UI에 과도하게 노출된다.
- 같은 요청을 반복했을 때 품질 성공률을 측정하는 평가 체계가 없다.

따라서 현재 문제는 프롬프트 문구만 보강해서 해결할 수 없다. 후보 생성, 검증, 승인, 조립을 포함한 생성 오케스트레이션을 다시 설계해야 한다.

## 범용 스프라이트 합격 기준

스프라이트 결과는 최소한 다음 조건을 만족해야 한다.

- 동일 캐릭터 정체성: 얼굴, 체형, 의상, 장비, 팔레트가 프레임 사이에서 유지된다.
- 동작 의미: idle은 미세한 대기 동작, walk는 접지와 보행 주기, attack은 준비·타격·회수 단계가 읽힌다.
- 시간 연속성: 정지 포즈 모음이 아니라 순서가 명확한 동작이다.
- 루프 가능성: loop 액션은 마지막 프레임에서 첫 프레임으로 자연스럽게 이어진다.
- 방향 일관성: 카메라 각도, 신체 방향, 장비 방향이 유지된다.
- 피벗 안정성: 발 기준점과 캐릭터 중심이 임의로 흔들리지 않는다.
- 규격 일관성: 프레임 크기, 알파, 여백, 행·열, 파일 순서가 선택한 출력 프로필과 일치한다.
- 부분 재생성: 실패한 액션·방향·프레임만 다시 만들 수 있다.
- 추적 가능성: 요청 레시피, reference, Provider, 모델, 후처리와 QA 결과를 manifest에 남긴다.

## 프로필별로 분리할 항목

다음 값은 제품 전체에 하드코딩하지 않고 버전이 있는 출력 프로필에 둔다.

- 프레임 W/H와 픽셀 밀도
- 방향 수, 이름과 순서
- 액션 목록, 프레임 수, FPS, loop 여부
- pivot, root, contact baseline
- 파일명과 디렉터리 구조
- 팔레트, 외곽선과 알파 규칙
- 미러 허용 방향쌍
- 엔진별 manifest와 importer 계약

예: `dungeon-cleanup-inc/actor-v1`은 88×88, 8방향 idle/walk를 정의하는 하나의 프로필일 뿐 범용 기본값이 아니다.

## 상태

- 사용자 확인: 범용 도트 게임 이미지 생성 프로그램이 목표다.
- 사용자 확인: 현재 수동 수정 기능보다 AI 생성 과정의 운 의존성과 메뉴 과잉이 더 큰 문제다.
- 사용자 확인: Hermes 에이전트를 기본 생성 경로로 유지한다.
- 사용자 확인: idle, walk, attack 등 스프라이트 동작 품질은 현재 불합격이다.
- 우선순위: 생성 파이프라인과 검증 체계 재설계가 P0다.
